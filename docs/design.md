# 阿姐终端

## 项目介绍

基于`NestJS`微服务架构模式(`monorepo`)、`一服一库`部署方式，以及`多租户(tenant)逻辑隔离`的设计方案，搭建统一身份平台系统。该系统平台作为底层基础，可为未来后续多硬提供接入点、`API`服务调用等服务功能。

简单来讲，就是统一认证授权，用以支持未来多应用接入和访问调用。

## 领域驱动设计

领域驱动设计，通过`战略设计(限界上下文，统一业务语言)`完成`业务拆分和粒度控制`；通过`战术设计(领域模型，领域服务，领域事件)`完成`强一致性的内聚`；通过`技术设计(可插拔组件)`完成`业务逻辑实现`。

特别注意，它是面向业务，不面向技术；因此，通过业务流程图(`行为命令->事件触发->确定事实`)，可以清晰地理解业务逻辑，从而进行领域驱动设计。

### 战略设计

#### 限界上下文

限界上下文(`BC`)，强调核心语义(`解决了什么问题`)，并界定职责边界。

<br/>

**拆分条件**

- 语义模型: 不同场景下的语义模型不一致；团队自治，服务独立部署；就得拆分界限上下文。

- 事件风暴: 行为命令，事件触发，确定事实；对强耦合的命令与事件聚类，画圈形成独立界限上下文。

<br/>

#### 统一业务语言

统一业务语言(`UL`)，强调业务术语(`在限界上下文内统一用语描述业务`)，避免鸡同鸭讲、减少沟通成本。

协作约定: `产品需求文档、话题讨论内容、程序编码变量、数据库表字段，要求四名合一，语义模型一致`，采用同一套业务语言进行业务沟通交流。

**统一语言词典**

| 业务中文 | 业务英文        | 业务定义与规则描述               | 所属限界上下文 |
| :------- | :-------------- | :------------------------------- | -------------- |
| 优惠券   | Coupon          | 运营发放的满减凭证，不可叠加。   | 营销上下文     |
| 满减券   | ReductionCoupon | 具体的券类型，必须关联商品分类。 | 营销上下文     |

### 战术设计

#### 领域模型

##### 聚合

聚合是一组强一致的生命周期绑定对象，通过聚合根保证不变量(`业务规则`)。内部实体对象对外不可见，聚合才是唯一对外的访问入口；因此，修改内部实体对象，就必须通过聚合根进行操作，以此保证生命周期的一致性。

特别注意: 当出现跨聚合操作，就应当通过消息事件进行解耦，并保证最终一致性。

<br/>

##### 实体

通过业务标识(`ID`)区分同一对象，具有完整的生命周期；只不过，作为实体，它的生命周期依附于聚合而存在。

特别注意: 当实体离开聚合，依然能保持完整的生命周期，此时可拆分为平级聚合；另外，实体发生变化，就会锁住与它相关联的所有聚合对象，在唱歌报告更大的并发瓶颈，因此实体必须独立成聚合。

<br/>

##### 值对象

通过值对象内容全等区分同一对象，不具有完整的生命周期。它包含业务规则，因此创建时必须自检数据输入的合法性。

<br/>

#### 领域服务

若领域行为明显归属于某个聚合，则定义在聚合根上；只有`当它不属于任何聚合根时，就应该考虑使用领域服务，并且仅限于只读、计算等操作`。

特别注意: 领域服务，常用于跨多个聚合对象来完成业务协作(`条件判定`、`计算结果`)，并非服务通信(`流程编排`、`消息事务`)。

<br/>

#### 领域仓储

领域仓储，用于抽象定义聚合根的操作行为(`库表数据的增删改查`)，但不涉及领域逻辑(`业务规则`)，只负责数据持久化(`数据入库`)。

简单来讲，就是定义数据库表操作的抽象接口；具体操作的实现，详见基础设施对应的持久层。

<br/>

#### 聚合工厂

聚合工厂，将复杂聚合对象的操作，封装创建过程中的业务规则和数据验证。

<br/>

#### 领域事件

领域事件，由聚合根产生已发生的业务事实，并通过最小化参数(`事件参数类接口`)描述业务事实，由订阅者在应用层或设施层响应，并决定如何执行副作用，且不改变已发生过的业务事实。

<br/>

#### 接口层

主要核心在于`协议适配`，不含有业务规则；总结概括为接收请求、协议转换、返回结果。

<br/>

#### 应用层

主要核心在于`用例编排`，不编写逻辑实现；总结概括为任务编排、事务边界、事件发布。

<br/>

##### 应用职责

| 维度          | 职责描述                   | 关键约束         |
| ------------- | -------------------------- | ---------------- |
| 用例编排​     | 实现业务用例，协调聚合根。 | 无业务规则。     |
| 事务边界​     | 管理事务的开启与提交。     | 不感知业务细节。 |
| 跨聚合协调​   | 协调多个聚合完成业务目标。 | 不持有领域状态。 |
| 输入输出适配​ | 将 DTO 转换为领域对象。    | 不返回领域对象。 |
| 事件发布​     | 将聚合根事件发布到外部。   | 不决定事件内。   |

<br/>

**用例定义**

```bash
用例名称：Change User Name
触发条件：操作员提交用户名变更请求
前置条件：
  - 用户必须存在
  - 用户状态必须为「活动（Active）」
业务规则：
  - 用户名在租户内唯一
  - 用户名格式符合规范
  - 冻结 / 废止状态的用户不允许更名
后置条件：
  - 用户名已更新
  - 产生 UserNameChanged 领域事件
```

<br/>

#### 领域层

主要核心在于`业务规则`，不依赖外部内容；总结概括为领域行为、业务规则、产出事实。

<br/>

##### 值对象职责

| 维度     | 职责描述                     | 关键约束   |
| -------- | ---------------------------- | ---------- |
| 等值判断 | 值全等即为同一对象。         | 值全等。   |
| 合法检验 | 创建时自动检查数据合法性。   | 创建自检。 |
| 业务规则 | 值非空、字符长度、匹配格式。 | 规则控制。 |

<br/>

##### 实体职责

职责边界: 断言自己、改变自己、封装自己、关联别人、计算自己。

| 维度      | 职责描述                               | 关键约束                        |
| --------- | -------------------------------------- | ------------------------------- |
| 状态断言​ | 主动防御，校验自身不变量(`业务规则`)。 | 纯函数，无副作用。              |
| 状态突变​ | 修改自身状态，守护业务规则             | 不触碰外部聚合。                |
| 内部封装​ | 管理内部属性，暴露最小接口。           | 保护不变量(`业务规则不暴露`)。  |
| 外部关联​ | 维护与外部聚合的。                     | 聚合`ID`级关联 不持有外部实体。 |
| 派生计算​ | 基于自身数据推导结果。                 | 不依赖外部状态。                |

<br/>

##### 聚合职责

职责边界: 守规则、管生死、排顺序、清引用、报事实。

| 维度          | 职责描述                         | 关键约束         |
| ------------- | -------------------------------- | ---------------- |
| 边界守卫​     | 校验跨实体规则，守卫聚合不变量。 | 纯函数，可复用。 |
| 生命周期管理​ | 控制聚合内对象的生、死、重组。   | 调用实体工厂。   |
| 内部编排​     | 协调多实体有序执行。             | 维护强一致性。   |
| 引用一致性​   | 清理孤儿引用，修复`ID`一致性。   | 不触碰外部状态。 |
| 事实发布​     | 产出领域事件，描述已发生事实。   | 驱动最终一致。   |

<br/>

#### 设施层

主要核心在于`技术实现`，不含有业务语义；总结概括为实现各层所定义的契约内容；尤其是领域层与应用层。

在设施层中，主要体现的是`技术实现`；每一个技术实现，都可以封装成`组件`，支持热插拔、可替换。

### 接口设计
